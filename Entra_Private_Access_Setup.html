Set Up Microsoft Entra Private Access

A practical field note for publishing private applications and RDP access through Microsoft Entra Private Access and Global Secure Access.

← Back to Security Field Notes
Entra ID Published note · anonymized field reference

Purpose

This note describes a practical setup pattern for Microsoft Entra Private Access to publish private applications, including RDP to internal servers, without exposing inbound access to the internet and without relying on traditional VPN access.

Design idea: users run the Global Secure Access client. Traffic for selected private apps is tunneled to Microsoft Entra Private Access, then sent through private network connectors to the internal resource.

What Entra Private Access is

Microsoft Entra Private Access is part of Global Secure Access. It gives users identity-aware access to private resources using Conditional Access and private network connectors. You can configure access broadly with Quick Access, or more granularly with per-app access.

Recommended model

Use Quick Access for initial testing or VPN-like access to a small private range. Use per-app access for production segmentation, especially for administrative protocols such as RDP.

Core components

Global Secure Access client
Installed on the user device. It identifies and forwards traffic that should go through Entra Private Access.
Private network connector
Windows Server connector installed inside the private network. It uses outbound connectivity to the Microsoft service.
Connector group
A logical group of connectors used to reach a site, datacenter, or application zone.
Quick Access
A simple way to define private FQDNs, IPs, or ranges that should be routed through Private Access.
Per-app access
A more controlled model where you define application segments, ports, users, groups, and Conditional Access policies.

Setup steps

  1. Confirm licensing and roles. Confirm that the tenant has the required Global Secure Access / Entra Private Access capability and that administrators have rights to configure Global Secure Access, enterprise apps, and Conditional Access.
  2. Enable Global Secure Access. In the Microsoft Entra admin center, open Global Secure Access and complete the initial activation steps.
  3. Install private network connectors. Install at least two connectors on Windows Server in the network that can reach the private resources. Avoid installing them directly on domain controllers unless there is a clear reason.
  4. Create connector groups. Group connectors by site, application zone, or network boundary. This makes it easier to control routing and troubleshoot access.
  5. Enable the Private Access traffic forwarding profile. Assign the profile to a pilot user group first.
  6. Deploy the Global Secure Access client. Start with a pilot group. For production, deploy with Intune or another software deployment tool.
  7. Configure Quick Access or per-app access. For a first test, define a private FQDN or IP. For production RDP, create a Global Secure Access app with the target segment and TCP 3389.
  8. Assign users and groups. Keep assignments narrow. Use dedicated admin groups for sensitive protocols such as RDP.
  9. Add Conditional Access. Require MFA, compliant device, trusted location rules if relevant, and session controls that match the risk of the app.
  10. Test and monitor. Test DNS resolution, TCP reachability, user assignment, Conditional Access result, connector health, and client traffic forwarding.

Example: publish RDP to an internal server

For administrative RDP, prefer per-app access instead of broad subnet access.

  1. Create or select a connector group that can reach the target server.
  2. Create a new Global Secure Access app.
  3. Add an application segment for the server FQDN or IP address.
  4. Add TCP port 3389.
  5. Assign only the admin group that should use this RDP path.
  6. Create a Conditional Access policy requiring MFA and a compliant or hybrid joined device.
  7. On the client, confirm the Global Secure Access client is connected.
  8. Start RDP using the server name or IP covered by the application segment.

Validation checklist

  • The connector status is active.
  • The connector server can resolve and reach the target application or server.
  • The user is assigned to the Private Access profile and the app.
  • The Global Secure Access client is installed, signed in, and forwarding Private Access traffic.
  • Conditional Access sign-in logs show the expected policy result.
  • RDP works only for the intended users and only through the intended path.

Common problems

  • DNS is not aligned. The client and connector path must resolve the same private resource correctly.
  • Connector placement is wrong. A connector must be placed where it can reach the target resource over the required port.
  • Too broad Quick Access ranges. This can become a VPN replacement with weak segmentation. Use per-app access for sensitive workloads.
  • Client is missing or not forwarding traffic. Private Access currently depends on the Global Secure Access client for end-user access.
  • RDP multi-session expectations. Check current known limitations before using this pattern for RDS, AVD, or shared jump-server designs.
Design warning: Entra Private Access is not just a network feature. Treat it as an identity-controlled access path. The value comes from user assignment, Conditional Access, device posture, logging, and segmentation.

Suggested pilot

  1. Start with one connector group and two connectors.
  2. Publish one low-risk internal web app using Quick Access.
  3. Then publish one RDP target using per-app access and TCP 3389.
  4. Assign five to ten admin users.
  5. Validate logs, user experience, support process, and rollback.
  6. Expand by application group, not by broad network range.
