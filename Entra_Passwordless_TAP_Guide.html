Going Passwordless in Microsoft Entra ID with TAP

A practical field guide for planning, piloting, and rolling out passwordless authentication using Microsoft Entra ID, Windows Hello for Business, passkeys, FIDO2 security keys, Microsoft Authenticator, and Temporary Access Pass.

Passwordless is not just a technical switch. It is an identity architecture decision, an onboarding process, a recovery model, and a user adoption project. The goal is not simply to remove passwords from the sign-in screen. The goal is to make passwords unnecessary in daily work and to move sensitive access to phishing-resistant authentication.

Field position: start with a controlled pilot, use Temporary Access Pass for onboarding and recovery, use Windows Hello for Business as the standard Windows experience, and require phishing-resistant methods for administrators and sensitive applications.

1. What passwordless means in Entra ID

In Microsoft Entra ID, passwordless means that users can authenticate without typing their account password. This can be done with several methods, and they do not all provide the same security level.

MethodBest suited forNotes
Windows Hello for BusinessCorporate Windows devicesStrong default method for Entra joined or hybrid joined Windows devices.
Passkeys / FIDO2 security keysAdministrators, high-risk users, regulated accessPhishing-resistant and suitable for privileged access.
Passkey in Microsoft AuthenticatorMobile-first passwordless scenariosUseful where a mobile authenticator is the preferred user experience.
Microsoft Authenticator passwordless phone sign-inGeneral workforceGood user experience, but evaluate against phishing-resistant requirements.
Certificate-based authenticationSmartcard or PKI-based environmentsPowerful, but more complex to operate.
Temporary Access PassOnboarding and recoveryBootstrap method. Not the long-term sign-in method.

2. Strategy and target state

The target state should be defined by user type and risk level. A single method for everyone rarely works well.

User typePrimary methodFallbackRequirement

Important: do not start by trying to remove every password immediately. Start by making password use unnecessary, then enforce stronger methods where the risk is highest.

3. How Temporary Access Pass works

Temporary Access Pass, usually called TAP, is a time-limited passcode created by an administrator. The user can use it to sign in and register a real passwordless method, such as a passkey, FIDO2 security key, Windows Hello for Business, or Microsoft Authenticator.

TAP solves the onboarding problem: users need to sign in to register passwordless methods, but the organization does not want them to depend on passwords during that registration.

Recommended TAP use: onboarding, lost device recovery, new phone registration, new security key registration, and controlled admin recovery.

Recommended TAP settings

SettingRecommendation
TargetStart with a pilot group. Do not enable broadly on day one.
Default lifetime1 to 8 hours for standard onboarding.
Admin lifetimeShorter duration, for example 15 to 60 minutes.
One-time useYes, as the default.
LengthUse at least 8 characters, preferably longer if the process allows it.
DeliveryUse a verified channel. Treat the TAP value as a secret.

4. Authentication methods to enable

Windows Hello for Business

Use Windows Hello for Business as the standard method for managed Windows devices. It gives the user a simple sign-in experience while keeping the private key protected on the device, preferably with TPM.

For hybrid environments, validate the on-premises access model. Cloud Kerberos trust is often the preferred modern model when users need access to traditional on-premises resources.

Passkeys and FIDO2 security keys

Use passkeys and FIDO2 security keys for administrators and high-risk users. For admin scenarios, prefer device-bound methods or physical security keys where the organization controls the lifecycle.

Microsoft Authenticator passwordless

Authenticator-based passwordless sign-in can be a good user experience for standard users. Make sure number matching, application context, and location context are enabled where applicable.

Certificate-based authentication

Use certificate-based authentication if the organization already has a strong PKI or smartcard model. It can be very strong, but it adds operational complexity.

5. Step-by-step implementation

Step 1: Assess current authentication methods

  1. Open the Microsoft Entra admin center.
  2. Go to Protection > Authentication methods.
  3. Review registration details and method usage.
  4. Identify users still depending on SMS, voice calls, or weak recovery methods.
  5. Identify administrators and high-risk users that should move first to phishing-resistant authentication.

Step 2: Clean up weak methods

  1. Review SMS, voice call, email OTP, OATH tokens, and security questions.
  2. Do not disable everything at once.
  3. Block weak methods for administrators first.
  4. Move standard users gradually after they have registered stronger methods.

Step 3: Enable Temporary Access Pass

  1. Go to Protection > Authentication methods > Policies.
  2. Open Temporary Access Pass.
  3. Enable TAP for a pilot group.
  4. Set short lifetimes and one-time use as the default.
  5. Document who may create TAP and how user identity must be verified.

Step 4: Enable passkeys / FIDO2

  1. Open the passkey or FIDO2 authentication method policy.
  2. Enable it for the pilot group.
  3. Decide whether synced passkeys, device-bound passkeys, and physical security keys are allowed.
  4. For administrators, prefer device-bound or physical security keys.
  5. Test registration, sign-in, lost key handling, and replacement.

Step 5: Enable Microsoft Authenticator passwordless

  1. Open the Microsoft Authenticator authentication method policy.
  2. Enable the method for the pilot group.
  3. Require number matching and context details.
  4. Test registration using TAP.
  5. Validate the user experience on both managed and unmanaged mobile devices.

Step 6: Configure Windows Hello for Business

  1. Use Intune to configure Windows Hello for Business.
  2. Require TPM where possible.
  3. Set a reasonable PIN policy.
  4. Allow biometrics if accepted by company policy.
  5. Test on Entra joined and hybrid joined devices before broad rollout.

6. User onboarding with TAP

A clean onboarding process reduces helpdesk load and prevents users from falling back to passwords or weak methods.

  1. Helpdesk or IAM verifies the user's identity.
  2. An administrator creates a single-use TAP with a short lifetime.
  3. The TAP is shared through an approved channel.
  4. The user opens https://aka.ms/mysecurityinfo.
  5. The user signs in with the TAP.
  6. The user registers the required method, such as passkey, FIDO2 key, Windows Hello for Business, or Authenticator passwordless.
  7. Helpdesk confirms that at least one approved passwordless method is registered.
  8. The user signs in again without using the account password.
Operational tip: create a short user guide with screenshots for the exact methods you support. Keep it separate from the admin runbook.

7. Recovery process

Passwordless fails operationally if recovery is weak. Plan for lost phones, replaced laptops, broken FIDO2 keys, forgotten PINs, and compromised accounts.

Standard user recovery

  1. Verify the user's identity according to the helpdesk process.
  2. Remove lost or compromised authentication methods.
  3. Create a short-lived, single-use TAP.
  4. Ask the user to register a new passwordless method.
  5. Check the sign-in logs if anything looks abnormal.

Administrator recovery

Administrator recovery should be stricter. Use stronger identity validation, approval, and logging. For highly privileged accounts, consider a two-person process before issuing TAP.

8. Conditional Access enforcement

Conditional Access should enforce the target state gradually. Start in report-only mode and move to enforcement when registration and support processes are ready.

9. Pilot and rollout plan

Phase 1: IT pilot

Start with IAM, security, endpoint management, and helpdesk. Validate TAP, passkeys/FIDO2, Windows Hello for Business, Authenticator passwordless, sign-in logs, and recovery processes.

Phase 2: Administrators and high-risk users

Move privileged users to phishing-resistant authentication. Remove SMS and voice fallback for admin accounts. Test admin portals, command-line tools, privileged workflows, and emergency access.

Phase 3: Business pilot

Select one or two business units with good support coverage. Measure registration success, failed sign-ins, helpdesk tickets, and user experience.

Phase 4: Broad rollout

Roll out by region, department, or device readiness. Use communication templates, manager briefings, and short user instructions.

Phase 5: Enforcement

When adoption is high and recovery is tested, enforce stronger Conditional Access policies and phase out weak methods.

10. Pre-enforcement checklist

  • Break-glass accounts exist and are monitored.
  • Administrators have at least two strong authentication methods.
  • TAP is enabled only for controlled groups.
  • Helpdesk has a documented identity verification process.
  • Users have clear onboarding instructions.
  • Windows Hello for Business has been tested on all supported device types.
  • Passkeys/FIDO2 have been tested for admin and recovery scenarios.
  • Legacy authentication is blocked or fully understood.
  • Conditional Access policies have been tested in report-only mode.
  • On-premises SSO requirements have been validated.
  • Lost phone, lost security key, and new laptop scenarios have been tested.
  • Sign-in logs and audit logs are reviewed during the pilot.

Recommended target state

The practical target state is simple: standard users should not need passwords for daily work, administrators should use phishing-resistant authentication, weak methods should be removed over time, and TAP should remain a controlled onboarding and recovery tool.

Final field recommendation: use TAP to get users into passwordless, not as a permanent sign-in method. Use Windows Hello for Business for managed Windows endpoints. Use FIDO2 or device-bound passkeys for administrators. Enforce the model with Conditional Access only after the pilot proves that onboarding and recovery work.
