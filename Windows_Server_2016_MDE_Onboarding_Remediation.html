Windows Server 2016 Defender onboarding remediation

Field note covering a Microsoft Defender for Endpoint onboarding failure caused by Windows component store corruption on Windows Server 2016.

Defender for EndpointWindows Server 2016DISMComponent Store

Overview

This case covers a Windows Server 2016 onboarding failure where the Microsoft Defender deployment tool repeatedly failed when enabling the built-in Windows Defender feature. The relevant blocker was not passive mode, policy, connectivity, or the onboarding package. The blocker was Windows servicing/component store damage.

ItemDetails
DeviceAnonymized Windows Server 2016 host
Operating systemWindows Server 2016 x64, build 14393
SymptomMicrosoft Defender deployment tool fails on every run. The Defender Antivirus feature will not enable.
Tool exit code710, EnableFeatureFailed
Observed patternMultiple failed runs with the same failure point and the same error signature.
Root causeThe Windows component store cannot finalize the Windows Defender feature because a referenced assembly is missing. DISM cannot self-repair without a valid servicing source.
Key point: Re-running the deployment tool, offboarding and re-onboarding, or changing passive mode settings will continue to fail until the Windows component store is repaired.

1. Evidence from logs

Deployment tool failure

The deployment tool completed prerequisite checks, downloads, telemetry, and connectivity checks. The recurring failure occurred when DISM attempted to enable the Windows Defender feature.

Failed to enable feature 'Windows-Defender', DISM exit code: 00003701
Code: EnableFeatureFailed. Failed to enable the 'Windows-Defender' feature, error: 0x00003701
Microsoft Defender deployment tool completed, exit code: 710 [EnableFeatureFailed]

Relevant DISM and CBS error codes

Error codeWhere seenMeaning
0x00003701 / 14081Defender deployment tooling logDISM exit code returned while enabling the Windows Defender feature. Microsoft documents this as 0x3701, meaning the referenced assembly could not be found.
0x80073701dism.log / CBSERROR_SXS_ASSEMBLY_MISSING. A component required by the install is missing from the component store.
0x800f081fdism.log during RestoreHealthThe source files could not be found. DISM /RestoreHealth could not locate a repair source.

Component store state

The parent Defender package was present, but the Defender feature and related sub-packages were stuck in a staged state instead of fully installed.

Package or featureCurrent stateTarget state
Windows-Defender-Server-Core-Package parentInstalledInstalled
Windows-Defender-FeaturesInstalledInstalled
Windows-Defender AV featureStagedInstalled
Server-Service, Core, NIS, and Management sub-packagesStagedInstalled

This sequence points to a Windows servicing issue. The Defender onboarding process cannot complete because the operating system cannot enable the required Defender feature.

2. Remediation sequence

Run the remediation from an elevated Command Prompt or PowerShell session. Take a backup or snapshot before starting and plan for at least one reboot.

Step 1. Provide a matching repair source

  1. Obtain a Windows Server 2016 install.wim or install.esd matching the server edition and build as closely as possible.
  2. Mount the ISO, for example as drive D:.
  3. Identify the correct image index inside the WIM.
Dism /Get-WimInfo /WimFile:D:\sources\install.wim

Step 2. Repair the component store from that source

Use /LimitAccess to force DISM to use the local media instead of attempting Windows Update again.

Dism /Online /Cleanup-Image /RestoreHealth /Source:wim:D:\sources\install.wim:<index> /LimitAccess
sfc /scannow

Confirm that DISM reports the restore operation as completed successfully. If this step fails, move to the escalation paths later in this note.

Step 3. Bring Windows servicing current

  • Install a current Servicing Stack Update for Windows Server 2016.
  • Install the latest applicable Cumulative Update.
  • Reboot after servicing updates.
  • If the LCU itself will not install, treat that as the same component store issue surfacing elsewhere.

Step 4. Enable the Defender feature manually

Dism /Online /Enable-Feature /FeatureName:Windows-Defender-Features /All
Dism /Online /Enable-Feature /FeatureName:Windows-Defender /All

Reboot, then confirm that the Defender feature is installed and that the windefend service can be queried.

Step 5. Re-run onboarding

  • Download a fresh deployment tool and onboarding package from the Defender portal.
  • Run onboarding again.
  • With the component store repaired, the EnableFeature step should complete and the deployment tool should exit with code 0 instead of 710.
  • Apply the latest Defender platform and security intelligence updates after onboarding.

4. If DISM repair still fails

If DISM /RestoreHealth fails even with a valid local source, the component store damage is deeper than a single missing payload. Work through the following options in order.

  1. Search CBS.log for the specific package named alongside 0x80073701. Download that exact package from Microsoft Update Catalog, expand it, and add it with Dism /Online /Add-Package. Then retry enabling the Defender feature.
  2. Try an alternate clean source image and repeat /RestoreHealth.
  3. Perform an in-place repair upgrade of Windows Server 2016 from matching or newer media while keeping apps, files, and settings.
  4. Open or continue a Microsoft support case for Windows Server and attach CBS.log, dism.log, and the Defender deployment tool log.

5. What not to do

  • Do not keep re-running the Defender deployment tool expecting a different result. It fails at the same DISM step.
  • Do not apply passive mode or active mode procedures to this server before the Defender feature installs. That addresses a different problem.
  • Do not hand-edit service Start values for windefend, wdfilter, wdboot, wdnisdrv, or wdnissvc under HKLM\SYSTEM\CurrentControlSet\Services. That can create an unsupported state.

6. Deployment at scale

Split the work by tool. Use BigFix, Configuration Manager, or another endpoint execution platform for component store repair, feature enablement, reboot handling, and onboarding. Use Intune only after onboarding for Defender policy management.

Important: Windows Server 2016 cannot enroll in Intune MDM. Intune can manage Defender security settings only after the server is already onboarded and healthy through Defender for Endpoint security settings management.

Source staging

At scale, the repair source becomes the main operational problem. Host matching Windows Server 2016 sources for the editions and installation types in the fleet, such as Standard vs Datacenter and Server Core vs Desktop Experience. Copy the WIM locally before running DISM to avoid reading multi-GB media across the WAN.

Detection logic example

$feat = (Dism /Online /Get-FeatureInfo /FeatureName:Windows-Defender) -match 'State :'
$svc = Get-Service windefend -ErrorAction SilentlyContinue
if ($feat -notmatch 'Enabled' -or -not $svc) { exit 1 }
exit 0

Remediation action example

$src = 'C:\Temp\WS2016_repair\install.wim'
$idx = 1
Dism /Online /Cleanup-Image /RestoreHealth /Source:wim:$src:$idx /LimitAccess
Dism /Online /Enable-Feature /FeatureName:Windows-Defender-Features /All /NoRestart
Dism /Online /Enable-Feature /FeatureName:Windows-Defender /All /NoRestart
# Reboot here, then run onboarding:
& 'C:\Deploy\DefenderDT.exe' -Quiet -Verbose -File:'C:\Deploy\WindowsDefenderATP.onboarding'

Reporting

Publish the Defender feature state, windefend service state, and last deployment tool exit code as endpoint management properties. This provides a fleet view of remediated, still failing with 710, and fully onboarded servers.

Rollout discipline

  • Start with a pilot ring of 5 to 10 servers.
  • Stagger reboots into maintenance windows.
  • Escalate servers where RestoreHealth still fails with a valid source.

7. Advanced recovery

In some cases, a plain Windows Server 2016 install.wim cannot repair the issue because the missing assemblies belong to cumulative updates already installed on the server. A base install.wim contains RTM files and cannot provide patched component versions.

Method A. Reinstall the exact updates named in CBS

Find the KBs named in CBS.log and reinstall the corresponding Windows Server 2016 x64 packages from Microsoft Update Catalog.

Dism /Online /Cleanup-Image /ScanHealth
Dism /Online /Get-Packages /Format:Table | findstr /i Pending
Dism /Online /Add-Package /PackagePath:C:\Repair\windows10.0-kbxxxxxxx-x64.msu

Reboot and retry enabling the Defender feature.

Method B. Repair from a patch-matched source

Create a repair source whose assembly versions match the server by servicing a mounted WIM with the same SSU and LCU level.

Dism /Mount-Image /ImageFile:D:\sources\install.wim /Index:<idx> /MountDir:C:\mount
Dism /Image:C:\mount /Add-Package /PackagePath:C:\Repair\SSU.msu
Dism /Image:C:\mount /Add-Package /PackagePath:C:\Repair\windows10.0-kbxxxxxxx-x64.msu
Dism /Online /Cleanup-Image /RestoreHealth /Source:C:\mount\Windows /LimitAccess
Dism /Unmount-Image /MountDir:C:\mount /Discard

Method C. In-place repair upgrade

If the component store is invalid and a consistency check is pending, the most reliable remediation is an in-place repair upgrade. Use Server 2016 media slipstreamed to the same or a higher build than the server, choose to keep apps and files, and allow setup to download updates.

Caution: Take a full backup or snapshot first. For domain controllers, review the supported repair upgrade guidance and ensure a system state backup exists.
