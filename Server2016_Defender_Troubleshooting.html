1. Executive summary

Main conclusion: The affected Windows Server 2016 device, server2016-app01, is onboarded to Microsoft Defender for Endpoint, but Microsoft Defender Antivirus is not active. The strongest evidence is PassiveMode = 2 and the Defender portal reporting Defender Antivirus mode: Disabled.

The issue is not caused by the assigned Defender AV policy itself. The policy reaches the server and many settings apply successfully. Settings that require the Defender AV engine fail because the AV engine is disabled or passive.

The most likely explanation is Windows Server 2016 legacy Defender behavior: Defender AV was placed into passive/disabled mode at some earlier point, possibly during onboarding or after a previous endpoint security product was present. Server 2016 does not reliably auto-reactivate Defender AV after passive mode conditions are removed.

6. Technical analysis

6.1 Preferences do not prove AV is active

Get-MpPreference shows configuration preferences, but it does not prove the Defender AV engine is active. On Server 2016, preferences and policies can exist while AV remains passive/disabled.

6.2 Server 2016 passive mode behavior

Windows Server 2016 behaves differently from newer Windows versions. If Defender AV is placed into passive/disabled mode, it may not automatically return to active mode when the original condition is gone. This can happen if another AV or security tool was detected at onboarding time or left remnants behind.

6.3 Why same onboarding can lead to different outcomes

  • Another AV may have been installed or registered at onboarding time.
  • An old security product may have left Windows Security Center registration behind.
  • The server may have been built from a different template.
  • The onboarding happened under a different Defender platform/version state.
  • Server 2016 does not reliably self-correct the mode later.

6.4 KACE discussion

Quest KACE itself is not an antivirus engine. However, if KACE deployed or previously managed third-party endpoint security, Windows may still have a registered AV product or remnants. That can keep Defender in passive mode on Server 2016.

7. Troubleshooting mode and tamper protection

Troubleshooting mode was enabled from the Defender portal. It is time-limited and allows some local Defender changes for investigation.

Earlier, editing the PassiveMode registry value failed with an error writing the new value, which indicated tamper protection or MDE self-protection was blocking local changes.

Later, Get-MpPreference showed:

DisableTamperProtection : True

This suggests that during troubleshooting mode, tamper protection was effectively relaxed or disabled for the troubleshooting window.

Clarification: There is no supported Set-MpPreference -DisableTamperProtection parameter. Troubleshooting mode can relax protections, but it does not make tamper protection a permanent locally managed setting.

9. Likely root cause

Most likely root cause: The server is stuck in Server 2016 Defender passive/disabled AV mode, indicated by PassiveMode = 2 and Defender portal showing Defender Antivirus mode: Disabled. The state was likely set during or before onboarding when Windows believed another AV/security product existed, or from an older onboarding/baseline condition.
  1. Server 2016 was onboarded to Microsoft Defender for Endpoint.
  2. At some point, Defender AV was placed into passive/disabled mode.
  3. Server 2016 did not automatically switch Defender AV back to active.
  4. The AV Enforcement Policy tried to configure Defender AV features.
  5. Settings that require the Defender AV engine failed.
  6. The portal reported the AV policy as Error and Defender Antivirus as not active.

10. Recommended next steps

10.1 Immediate diagnostics

  1. Run Get-MpComputerStatus to confirm current AV running mode.
  2. Run the WSC antivirus product query to confirm no ghost/leftover AV registration.
  3. Check Defender operational event log for service start or AV mode errors.
  4. Compare outputs with one working Server 2016 machine in the same site/group.

10.2 If no other AV is registered and AV remains disabled

  1. Offboard the server from Microsoft Defender for Endpoint.
  2. Reboot.
  3. Ensure no third-party AV remnants or WSC registrations exist.
  4. Ensure Defender AV components are installed and healthy.
  5. Re-onboard using the current supported Defender for Endpoint onboarding method for Windows Server 2016.
  6. Verify AMRunningMode, AntivirusEnabled, and policy status.

11. Suggested customer-facing wording

The issue on SERVER2016-APP01 is not caused by a broken AV policy or by GPO. The server is receiving the Defender AV policy, but Defender Antivirus itself is not active. The local registry shows PassiveMode = 2 and the Defender portal reports Defender Antivirus mode as Disabled. On Windows Server 2016, Defender AV can remain in passive/disabled mode after onboarding or after another security product was detected, and it does not always automatically switch back to active mode later.

The recommended remediation is to verify no other AV product is still registered, then offboard and re-onboard the server using the current supported Defender for Endpoint onboarding method for Server 2016. After that, Defender AV should become active and the assigned AV enforcement policy should apply successfully.
