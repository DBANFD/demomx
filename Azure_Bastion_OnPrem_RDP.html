Azure Bastion to On-Prem RDP

A practical field note for using Azure Bastion IP-based connection to reach an on-premises Windows host over private network connectivity.

← Back to Security Field Notes
Azure Security Published note · anonymized field reference

Purpose

This note describes how to use Azure Bastion to reach an on-premises Windows host over RDP without exposing RDP to the internet and without building a traditional jump server.

Design idea: the admin connects to Azure Bastion, Bastion targets the private IP of the on-premises server, and traffic flows over VPN or ExpressRoute.

When this makes sense

  • You already have hybrid connectivity between Azure and the on-premises network.
  • You want controlled administrative RDP access without public IPs.
  • You want to avoid VPN client rollout for administrators.
  • You are comfortable using Bastion as a controlled jump point.

High-level flow

  1. User opens Azure Bastion in the Azure portal or uses the native client option.
  2. Bastion connects to the target by private IP address.
  3. The traffic is routed through the Azure virtual network to the on-premises network over VPN or ExpressRoute.
  4. The target server only needs to allow RDP from the expected network path.

Prerequisites

Azure Bastion SKU
Use Standard or higher for IP-based connection.
Hybrid connectivity
Site-to-site VPN or ExpressRoute private peering must exist between Azure and the on-premises network.
Routing
The Bastion virtual network must be able to route to the on-premises server subnet.
Firewall
Allow TCP 3389 from the expected Bastion path to the Windows host.
Identity
Use local or domain credentials for the RDP session.

Setup steps

  1. Deploy or upgrade Azure Bastion. Place Bastion in the virtual network that has the hybrid route to the on-premises network.
  2. Enable IP-based connection. In the Bastion configuration, enable IP-based connection and wait for the resource to update.
  3. Validate routing. Confirm the on-premises server subnet is reachable from the Azure network path used by Bastion.
  4. Connect by private IP. Open the Bastion resource, select the IP-based connection option, enter the private IP, choose RDP, and connect.
  5. Restrict access. Use RBAC, network controls, and administrative process controls so Bastion does not become an unmanaged shared jump path.

Native client example

When native client support is enabled, the Azure CLI can start an RDP session through Bastion to a private IP.

az network bastion rdp \
  --name <bastion-name> \
  --resource-group <resource-group> \
  --target-ip-address 10.20.30.40

Common problems

  • Routing is wrong. The most common issue is that the Bastion virtual network does not have a valid path to the target subnet.
  • Forced tunneling/default route breaks the design. Be careful with advertised default routes over ExpressRoute or VPN.
  • Firewall rules are too narrow or too broad. Allow the required path, but avoid opening RDP broadly across internal networks.
  • Expecting Entra ID login for IP-based RDP. For IP-based RDP, plan for local or domain credentials.
Design warning: this is still RDP. Bastion reduces exposure, but it does not remove the need for privileged access control, logging, patching, and least privilege.
